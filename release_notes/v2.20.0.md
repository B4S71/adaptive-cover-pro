# v2.20.0

## 🔑 Highlights

Venetian blinds on KNX, Somfy IO/Tahoma, Shelly 2PM, and similar dual-axis hardware have been a persistent pain point: running two paired instances (one `cover_blind`, one `cover_tilt`) against the same physical entity caused the tilt instance to read motor back-rotate as a manual override every time the position instance moved the carriage. The two instances fought each other on a 5-minute cycle. v2.20.0 ships a dedicated `cover_venetian` sensor type that owns both axes from one config entry, sequences position-then-tilt commands so the motor finishes before the slat command lands, and absorbs back-rotate drift in a 90-second suppression window. Eleven beta cycles drove out the edge cases across KNX, Somfy, and Zigbee hardware — this stable release is the consolidated result of that work.

Three additional improvements landed alongside the venetian work: an `is_sunny` binary sensor option for cloud suppression, a fix for custom-position sensors that got stuck at slot position on release, and clearer display of held position during manual override.

## ✨ Features

- **New `cover_venetian` sensor type — dual-axis venetian blind support (#33):**
  Select "Venetian Blind (Dual-Axis)" in the setup wizard. The geometry screen
  captures window dimensions and slat properties in a single step. One config
  entry drives both `set_cover_position` and `set_cover_tilt_position` on the
  same HA cover entity.

  - **Position-then-tilt sequencing:** `CoverCommandService` sends
    `set_cover_position` first, polls `current_position` every 500 ms until the
    cover settles (within tolerance, or no progress for three consecutive idle
    samples, hard-capped at 60 s), then sends `set_cover_tilt_position` with the
    engine-derived angle. The motor finishes its back-rotate before the tilt
    command lands, eliminating the ping-pong fight.

  - **Per-axis manual override with tilt suppression:** Tilt drift that arrives
    within a 90-second window after a position command is recorded as
    `manual_override_rejected_tilt_suppression` and ignored — it's motor
    back-rotate, not user input. Tilt changes that arrive outside that window still
    trip manual override on the tilt axis. Position-axis override behavior is
    identical to `cover_blind`.

  - **Solar gate:** Tilt commands fire only when the pipeline's SOLAR handler is
    active. Overrides, manual holds, weather suppression, and other pipeline
    outcomes suppress tilt entirely. Covers stay quiet when they're not
    sun-tracking.

  - **Tilt-only mode:** The `venetian_mode` option (`position_and_tilt` default,
    or `tilt_only`) skips the position command and sends only
    `set_cover_tilt_position` each cycle — useful when vertical position is managed
    by a separate automation or when covers like Shelly 2PM are driven tilt-first.

  - **Continuous tilt tracking:** Tilt is recalculated and dispatched on every
    update cycle, not only when the resolved position changes. A cloud passing
    overhead can update position without moving it; tilt now tracks the sun angle
    independently.

  - **Configurable retraction guards:** "Skip Tilt Above Position (%)" (default
    95%) prevents tilt commands when the slats are retracted into the housing and
    tilting is physically meaningless. An upper threshold of 95% means the slat
    command is skipped when the carriage is above that point.

  - **Maximum and Minimum Slat Tilt:** `max_tilt` (0–100, default 100) caps the
    tilt angle the engine will ever command — useful for hardware that creates
    glare at full close. `min_tilt` (0–100, default 0) floors the angle — useful
    for blinds that block all light at 0%. The engine clamps the computed tilt to
    `[min_tilt, max_tilt]` before dispatch.

  - **Tilt delta gate:** Tilt commands respect the same Min Change threshold
    (`delta_position`) as position commands. When the target tilt is within
    `min_change` percent of the last-sent tilt, the sequencer skips the
    `set_cover_tilt_position` call and emits a
    `tilt_command_skipped(delta_too_small)` diagnostic event. Prevents chatty tilt
    updates when the sun moves only slightly between cycles.

  - **Inverse tilt (`inverse_tilt` option):** Flips the tilt axis for hardware
    wired opposite to the HA convention (0 = slats open, 100 = slats closed). The
    integration stores the logical value internally and converts to the wire value
    only when sending the command. Distinct from the existing Inverse State option,
    which controls the carriage axis.

  - **Target Tilt sensor:** A new diagnostic sensor exposes the engine's target
    tilt angle (in degrees) in the HA entity list alongside the existing Target
    Position sensor.

  - **Decision-trace tilt:** The Decision Trace sensor and diagnostics download now
    show the tilt the engine derived alongside the resolved position. A synthetic
    `venetian_engine` terminal step records the slat-angle computation.

  - **Tilt diagnostic events:** All tilt command outcomes emit EventBuffer events:
    `tilt_command_sent`, `tilt_command_skipped`, `tilt_command_verified`,
    `tilt_command_drift`. The diagnostics event timeline covers both axes.

- **`is_sunny` binary sensor for cloud suppression (#363):**
  An optional binary sensor, input boolean, switch, or schedule can now drive the
  `is_sunny` flag consumed by `CloudSuppressionHandler`. Useful when your weather
  integration doesn't expose a `condition` attribute or when you want a custom
  sunny signal from your own automations. When the sensor is unavailable the
  integration falls back to the existing weather-entity logic. Installs that don't
  configure the sensor see no behaviour change.

- **Target Position shows held position during manual override (#33):**
  `ManualOverrideHandler` now populates `held_position` on the pipeline result from
  the cover's actual current position. The Target Position sensor and diagnostics
  download show where the cover physically sits during override. A cover held at
  100% by the user no longer shows a target of 20%.

- **Cloud suppression master toggle moved to top of config step (#364):**
  The cloud suppression on/off toggle now appears before the trigger sensors it
  gates in the Light & Cloud configuration step. Previously the toggle was buried
  below the sensor fields, which led users to configure sensors before realising
  the feature was off. The label and help text are reworded so it is clear the
  field enables or disables cloud suppression as a whole.

## 🐛 Bug Fixes

- **User-supplied name overwritten in cover entity setup step (#362):**
  `async_step_cover_entities` unconditionally derived the entry name from the
  first selected cover's `original_name`, overwriting any name the user had
  already typed in the earlier step. In setups where multiple source covers share
  a generic `original_name` (common with Tuya curtain modules) this collapsed
  distinct entries toward the same generated title. The guard now skips
  auto-generation when a name is already present in the flow config.

- **Custom-position sensor release stuck at slot position (#365):**
  When a custom-position sensor turned off, the time-delta gate blocked the return
  move and the cover stayed stuck at the slot's position until the gate elapsed.
  The fix tracks per-sensor on/off state across cycles and injects `force=True`
  with reason `custom_position_released` when the released sensor was the active
  trigger, mirroring the existing force-override release pattern. The workaround
  of toggling Automatic Control off then on is no longer needed.

- **False manual override on assumed-state covers:**
  Covers exposed via integrations that report `assumed_state=true` (some KNX,
  MQTT, and template covers) don't drive real state-change events when HA
  dispatches a command. `PositionContextTracker` now mints an explicit `Context`
  for every `set_cover_position`, `open_cover`, and `close_cover` call and records
  its id. State-change events whose context id is absent from the tracker and whose
  `user_id` is non-null are marked as manual immediately, before the position-math
  path can be defeated by post-reconcile state.

- **False manual override on slow Zigbee covers (#361):**
  Some Zigbee covers (e.g. ZHA Rollershade) only report `current_position` at the
  end of motion. When motion took longer than the 45 s transit timeout, the first
  state-change event arrived mid-traverse with a stale position; the position-math
  check then misread the gap as a manual nudge. `handle_state_change` now records
  `manual_override_rejected_in_transit` and returns early when the cover's own
  state attribute is `opening` or `closing` at event time, deferring the check to
  the settling event with the true final position. Tilt-axis changes during a
  position traverse are unaffected.

- **False manual override on tilt-only command path (#33):**
  When the cover was already at the commanded position, `update_tilt_only` sent
  `set_cover_tilt_position` without stamping the back-rotate suppression window.
  Each tilt-axis state-change event during slat settle produced a burst of spurious
  `manual_override_set` events. `_send_tilt_command` now stamps the suppression
  window before calling `set_cover_tilt_position`, so both the post-settle and
  tilt-only paths block spurious override trips during settle.

- **Shelly 2PM false-settle on position poll (#33):**
  The position settle poll was counting idle samples without checking whether the
  cover was still in motion. Shelly 2PM covers publish position at roughly
  1-second intervals; three equal readings in a row triggered "settled" while the
  motor was mid-travel, firing the tilt command too early. The stall counter now
  only increments when `cover.state` is not opening or closing.

- **Tilt drift stuck on stale target (#33):**
  After a tilt command, if the reported tilt drifted more than `VENETIAN_TILT_VERIFY_TOLERANCE`
  (5%) from the commanded angle, the coordinator previously treated the stale
  target as satisfied and stopped retrying. The fix clears the target when drift
  exceeds the tolerance, forcing a retry on the next cycle.

- **Tilt verification broken under inversion (#33):**
  The post-command drift check was comparing the wire reading directly against the
  logical target. With `inverse_tilt` enabled this produced a ~100% apparent delta
  on every cycle, clearing the target and forcing a retry every update. The fix
  applies the inversion transform to the wire reading before the delta comparison,
  so the check operates entirely in logical space.

- **Two cooperating manual override bugs on new venetian installs (#33):**
  `SecondaryAxisCheck.evaluate` was short-circuiting to `consumed=False` when
  `new_value == expected`, bypassing the suppression callback and letting
  position-axis drift reach the manual-override handler. Simultaneously,
  `_rebase_commanded_position` captured `current_position` before the motor's
  back-drive settled, so the rebase was skipped and the commanded target drifted
  from the actual settled position. Covers behaving normally were tripping manual
  override as a result. The fix checks suppression before the on-target early exit
  and adds a 1.5-second settle delay before the rebase.

- **Back-rotate drift false override after suppression window (#33):**
  When back-rotate drift arrived after the 90 s suppression window closed,
  `SecondaryAxisCheck` was letting the event propagate unconsumed into the
  manual-override handler. Setting `consumed=True` on the suppression-rejection
  path closes that gap.

- **Boundary drift false override at position tolerance floor (#33):**
  The position check used a strict `<` comparison, so drift exactly equal to the
  effective tolerance floor fell outside the absorbed range and incorrectly tripped
  manual override after a venetian tilt back-drive. The comparison is now `<=`,
  matching the documented semantics.

- **Post-tilt reconciliation loop (#33):**
  After `set_cover_tilt_position` fired, the coordinator was treating each
  subsequent `current_tilt_position` state change as a new deviation and
  re-sending the tilt command. The reconciliation loop is now in-flight-aware and
  skips re-sends while the motor is settling.

- **Actuator tilt cache reasserts slats on fully-retracted blind (#33):**
  KNX and Shelly actuators retain their last commanded tilt internally and reapply
  it ~1–2 seconds after the carriage settles. Without overwriting that cache when
  the carriage retracts above `tilt_skip_above`, the prior solar-cycle tilt
  reasserted and closed the slats on a fully-retracted blind. `VenetianPolicy.after_position_command`
  now sequences a neutral tilt command (`POSITION_OPEN`) after the carriage settles
  above the threshold, overwriting the actuator cache with a known neutral value.

## 📖 Documentation

- **New [Venetian Blinds] wiki page:** Covers geometry inputs, sequenced-command
  timing, supported integrations (KNX, Somfy IO, Shelly 2PM), and the migration
  path for users currently running paired `cover_blind` + `cover_tilt` instances
  against the same entity.

- **HACS default store installation:** The quickstart now directs users to add the
  integration via the HACS default store. The separate custom-repository step has
  been removed from the docs (commit aa65aaa).

## 🧹 Chores

- **`CoverTypePolicy` architecture:** Venetian-specific geometry schema,
  entity-selector domains, and glare-zone support are gated through a
  `CoverTypePolicy` interface, eliminating cover-type string branches from the
  coordinator and config flow.
