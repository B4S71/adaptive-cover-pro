# v2.22.0-beta.5

beta.5 is a single targeted fix for issue [#33]. beta.4 (PR [#406]) suppressed false manual-override on the venetian tilt axis during the command-grace window; PR [#408] addresses two cooperating defects that remained on slow-start actuators like Somfy IO via Tahoma: a premature stall declaration in the position-settle loop, and a publish-lag window that started from the wrong clock. Together they caused the back-rotate cap to reassert ~22 s before the cover actually stopped moving, leaving a real back-rotate burst 27 s post-settle unprotected and visible to the manual-override path.

## 🎯 Should I install this?

- **If you run venetian blinds on a Somfy IO (Tahoma/Connexoon) actuator and saw manual override trip after a position command**, install this. The two defects fixed here were only reproducible on actuators with 3–5 s motor startup delays.
- **If you were on beta.4 and it was working**, this update is low-risk and resolves the one remaining edge case in [#33].
- **If you are on v2.21.5 without any active issue**, stay there. This beta exists to close out issue [#33] before the v2.22.0 stable release.

## 🔑 What's new in beta.5

### 🐛 Bug Fixes

- **Venetian publish-lag window anchored to settle transition + startup grace ([#33], PR [#408], commit 32851d3):**

  Two cooperating defects on Somfy IO and other slow-start actuators:

  **Defect 1 — premature stall declaration:** `_wait_for_position_settle` counted three consecutive unchanged-position samples as a stall and declared settle before the motor had started moving. Somfy IO takes 3–5 s to begin physical travel after the service call; during that window `cover.state` still reads `open`/`closed` and position is unchanged, so the stall counter fired ~22 s before the cover actually stopped. The fix adds a `VENETIAN_POSITION_SETTLE_STARTUP_GRACE_SECONDS = 6.0` constant: stall declaration is now blocked until either the cover has been observed in a moving state at least once, or the startup-grace window has elapsed. The existing 60 s deadline (`VENETIAN_POSITION_SETTLE_TIMEOUT_SECONDS`) still hard-caps a dead motor.

  **Defect 2 — publish-lag clock anchored to wrong timestamp:** `is_in_suppression_with_cap` previously bypassed the 30% back-rotate delta cap for 5 s after `stamp_position_command` (`VENETIAN_POST_SETTLE_CAP_GRACE_SECONDS`). On Somfy IO, the premature settle stamp started that 5 s clock 22 s early, so when the real back-rotate burst arrived 27 s post-stamp the cap had already reasserted. The fix adds a third tier: a `VENETIAN_BACKROTATE_PUBLISH_LAG_SECONDS = 45.0` window anchored to the actual `moving → settled` transition observed by the sequencer (not to `stamp_position_command`). The sequencer records this transition via a new `_stamp_settled` method, which `run_sequence` writes after `_wait_for_position_settle` returns and `is_in_suppression_with_cap` lazy-writes on the first non-moving cap query. `stamp_position_command` clears `_settled_at` at the start of each cycle so a prior-cycle settle cannot leak its window forward.

  The resulting three-tier suppression model in `is_in_suppression_with_cap`:

  - **(a) Carriage mid-travel** (`cover.state` in `opening`/`closing`): any delta suppressed.
  - **(b) Command-grace tail** (5 s after `stamp_position_command`): large deltas still suppressed — catches fast actuators (KNX, Shelly 2PM) whose back-rotate burst lands just after settle.
  - **(c) Publish-lag window** (45 s after the observed `moving → settled` transition): large deltas still suppressed — catches slow-bus republish on Somfy IO, KNX/Z2M where the back-rotate tilt value lands tens of seconds after the cover physically stops.

  Outside all three tiers the legacy geometry-bounded cap applies: a delta above `VENETIAN_BACKROTATE_MAX_DELTA_PERCENT` (30%) is a user move and manual override latches normally.

  Auto Control off→on resets now also clear `_settled_at` so a stale settle stamp from a prior command cycle cannot extend its window into the next.

## 📖 Cumulative scope (v2.22.0 series)

beta.5 is cumulative. Everything in [v2.22.0-beta.1] through [v2.22.0-beta.4] is included:

- **Nine-phase internal refactor (Phases A–H, beta.1/beta.2):** Coordinator responsibility distributed into `managers/`, `state/`, `pipeline/`, and `cover_types/`. Coordinator is ~640 lines lighter vs v2.21.5. Manual-override detection in `managers/cover_command/state_classifier.py`; cover-type-specific behaviour on `CoverTypePolicy` subclasses; cover-type literal branching eliminated. Net diff for the refactor: 62 files, +3,682 / −1,251 lines.

- **Post-settle cap grace — false venetian override fix ([#33], beta.2):** `DualAxisSequencer.is_in_suppression_with_cap` holds the cap bypass active for 5 seconds after a position command stamp, preventing KNX, Shelly 2PM, FGR223, and similar actuators from tripping false manual-override when `cover.state` settles before the tilt-walk burst finishes publishing.

- **Phase I — config-flow plumbing refactor (beta.3):** Collapsed duplicated schema reads, step routing, and options-step boilerplate into shared helpers, reducing `config_flow.py` by 168 lines.

- **Legacy climate step + optional-key cleanup (beta.3):** Removed `async_step_climate` and `CLIMATE_SCHEMA` dead code; extracted optional-key constants for `WEATHER_OVERRIDE`, `LIGHT_CLOUD`, and `TEMPERATURE_CLIMATE` schemas to prevent [#377]-class drift; fixed a latent omission where `CONF_CLOUDY_POSITION` was missing from its inline optional-key list.

- **"Is Sunny" sensor can now be cleared in the options flow ([#377], beta.3):** `CONF_BINARY_SENSOR` added to the optional-entities list so a blank submission removes the prior value instead of silently keeping it.

- **MODE2 tilt climate handler: sun-side awareness + `min_position` warning ([#373], beta.4):** The climate handler now picks the correct MODE2 hemisphere from the sign of gamma; `TiltPolicy.climate_tilt_percentage` is the single conversion helper for all four call sites. A ⚠️ warning surfaces in the config-flow summary when MODE2 is active and `min_position` ≥ 50.

- **Venetian tilt-axis command-grace suppression ([#33], beta.4):** `VenetianPolicy.secondary_axis_check` now carries a suppression callback that OR-composes the back-rotate window with a `_is_in_tilt_command_grace` helper, closing the last false-override vector on the tilt axis during a command-grace window.

- **Venetian publish-lag anchored to settle transition + startup grace ([#33], beta.5, above).**
