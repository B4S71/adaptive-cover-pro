# v2.22.1-beta.4

beta.4 is a forecast-reliability release: it closes five cooperating defects that kept `sensor.<cover>_position_forecast` frozen after boot, sharpens FOV-event timing from ┬▒15 min to ┬▒5 min, adds two performance improvements that reduce the per-forecast cost for multi-cover instances, and consolidates the integration's enum definitions into `const.py`.

## ­¤Ä» Highlights

- **Frozen forecast fixed.** Three independent bugs each silently prevented the 5-minute recompute from reaching the sensor: tasks garbage-collected before their first `await` (#448), ticks dispatched to a worker thread instead of the event loop (which caused `loop is not the running loop`), and a missing `async_update_listeners()` call that left the data update stranded in `coordinator.data`. All three are fixed.
- **FOV event accuracy ┬▒5 min.** The "Sun enters FOV" and "Sun exits FOV" event markers in the forecast are now refined against SunData's native 5-minute grid, down from ┬▒15 min (one full forecast step).
- **Shared SunData cache across entries.** Instances at the same location now share a single astral + `pd.date_range` fill per day instead of each paying the full cost independently (#441, #445).
- **O(1) forecast index lookup.** `_nearest_index` in `forecast.py` replaces the prior linear scan with a direct arithmetic calculation, keyed on `SUN_DATA_STEP_SECONDS = 300` (#442, #447).
- **Recorder footprint reduced.** Five sensors now declare `unrecorded_attributes` so HA's recorder stops persisting large array attributes (`forecast`, `events`, `trace`, `custom_position_slots`, `actual_positions`, and others) that the Lovelace card reads live but nobody queries from history (#446).

## ­¤Éø Fixed

### Forecast tick chain ŌĆö five cooperating fixes

`sensor.<cover>_position_forecast` was frozen at its boot-time value in all but the lightest-activity installations. The root causes formed a chain:

- **Tasks destroyed before first `await` (#448, commit `5b8045b`):** `hass.async_create_background_task` was called from a sync timer callback, where the task reference is weakly held and can be collected by the GC before the event loop runs it. The symptom was repeated "Task was destroyed but it is pending!" entries in the HA log. The fix switches both the initial-compute task and the periodic tick to `config_entry.async_create_background_task`, which pins each task to the entry's lifetime.

- **Tick function dispatched to executor (`f6fe3b0`):** Without `@callback`, HA classifies the plain `def _tick` as `HassJobType.Executor` and runs it in a worker thread. From there, `loop.create_task(..., eager_start=True)` raises `RuntimeError: loop is not the running loop`, so the task is never created and the recompute silently does nothing. Adding `@callback` to `_tick` keeps it on the event loop where it belongs. This was the root cause of the freeze on live instances where #448 alone did not restore updates.

- **Ticks staggered per-entry (`3e217e0`):** The scheduler used `async_track_time_interval`, which fires each entry's tick at a fixed offset from its own startup time. Switching to `async_track_time_change` with `minute=range(0, 60, FORECAST_RECOMPUTE_INTERVAL_MIN), second=0` aligns every entry to the wall-clock 5-minute grid (`:00`, `:05`, `:10`, ŌĆ”), so the dashboard sees a single synchronised refresh.

- **Listeners not notified after recompute (`c41c27e`):** `async_recompute_forecast` updated `coordinator.data.position_forecast` via `dataclasses.replace` but never called `async_update_listeners()`. The sensor only got a fresh value on the next regular coordinator cycle ŌĆö up to 4ŌĆō20 minutes later depending on activity. The fix adds `self.async_update_listeners()` immediately after the data field is updated, gated on `self.data is not None` to skip the pre-first-refresh window.

- **FOV-enter/exit events lag the position curve (`03e5dab`):** The event marker for `fov_enter` / `fov_exit` was placed at the first 15-minute forecast sample whose handler had already flipped, which could lag the actual FOV crossing by up to a full step. The new `_refine_fov_crossing` function in `forecast.py` scans SunData's native 5-minute grid between the two bracketing samples and places the event at the first grid point where `direct_sun_valid` actually flips, reducing timing error from ┬▒15 min to ┬▒5 min.

## ŌÜÖ’ĖÅ Changed

- **O(1) forecast index lookup (#442, #447, commit `6045d1a`):** `_nearest_index` in `forecast.py` replaces a linear scan over ~289 items with a single arithmetic expression: `max(0, min(len(times) - 1, round(delta / step_seconds)))`. The step is parameterised via `SUN_DATA_STEP_SECONDS = 300` in `const.py` so the function stays correct if the cadence changes. The previous linear scan was also slightly incorrect: it used a separate `target_tz` variable rather than coercing `target` in-place; the rewrite fixes both issues together.

- **Shared SunData day cache (#441, #445, commit `412c4ef`):** `sun.py` gains a module-level `_DAY_CACHE: dict[tuple, _SunDayData]` keyed on `(timezone, lat, lon, elevation, date.today())`. All `SunData` instances at the same location share one `pd.date_range` + astral fill per day; a `threading.Lock` guards concurrent fills via double-checked locking. The instance fast path (existing `_cache_day` check) is unchanged as tier 1; the module dict lookup is tier 2 (no lock needed under CPython); under-lock fill is tier 3. Stale (non-today) entries are purged when a new day's entry is stored.

- **Recorder exclusion for heavy sensor attributes (#446, commit `6e19451`):** `_SensorSpec` gains an `unrecorded_attributes: frozenset[str]` field. A resolver in `sensor.py` composes per-spec subclasses so HA reads `_unrecorded_attributes` at class init. Excluded keys: `actual_positions`, `actual_distances`, `position_explanation` (Cover Position sensor); `manual_covers` (control_status); `trace`, `custom_position_slots`, `enabled_handlers` (decision_trace); `forecast`, `events` (position_forecast); `per_entity` (position_verification). Live state, templates, automation triggers, and the Lovelace card are unaffected ŌĆö only the recorder DB stops persisting these keys.

- **Enums consolidated into `const.py` (`b9c2e53`):** `enums.py` is deleted. `CoverType`, `TiltMode`, `TemperatureSource`, `PresenceDomain`, `ClimateStrategy`, and `ControlMethod` move into a new "Enumerations" section (section 27) of `const.py`. `SensorType` is replaced by the proper `CoverType(StrEnum)` and all importers are updated. Forecast constants (`FORECAST_STEP_MINUTES`, `FORECAST_WINDOW_HOURS`, `EVENT_SUNRISE`, `EVENT_SUNSET`, `EVENT_FOV_ENTER`, `EVENT_FOV_EXIT`) move from `forecast.py` into `const.py` section 8a. No behavior change.
